# Git ワークフロールール

## ブランチ戦略

### ブランチ命名規則

| プレフィックス | 用途 | 例 |
|-------------|------|-----|
| `feature/` | 新機能の開発 | `feature/#42-user-auth` |
| `fix/` | バグ修正 | `fix/#123-login-error` |
| `docs/` | ドキュメントのみの変更 | `docs/#45-api-spec` |
| `refactor/` | リファクタリング | `refactor/#67-extract-utils` |

- ブランチ名には Issue 番号を含める
- 英語の小文字とハイフンで記述

### ブランチ作成前の必須手順

1. メインブランチ（main または master）に移動する
2. `git pull` でソースを最新化する
3. 最新の状態からブランチを作成する

```bash
git checkout main
git pull
git checkout -b feature/#42-user-auth
```

## コミットルール

### コミットメッセージ

- **日本語で記述する**
- 変更内容を簡潔かつ具体的に記述する
- Issue 番号がある場合は含める

```
良い例:
  #42 ユーザー認証のログイン機能を実装
  #123 ログイン画面のバリデーションエラーを修正
  APIレスポンスの型定義を追加

悪い例:
  fix
  update
  修正
```

### コミット単位

- 1つの論理的な変更を1コミットにまとめる
- 関連しない変更を1コミットに混ぜない

## プルリクエスト

### PR 作成ルール

- タイトルと説明は日本語で記述する
- 関連する Issue を連携する（`Closes #番号`）
- 変更内容の概要と影響範囲を記述する

### PR テンプレート

```markdown
## 概要
[変更内容の概要]

## 関連 Issue
Closes #番号

## 変更内容
- [変更1]
- [変更2]

## テスト
- [ ] テストを追加・更新した
- [ ] 既存テストが通ることを確認した
```
