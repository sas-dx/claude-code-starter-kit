# テストルール

## 基本方針

### TDD（テスト駆動開発）

1. **テストを先に書く** - 実装コードの前にテストを作成する
2. **テストを通す実装を書く** - テストが通る最小限のコードを実装する
3. **リファクタリング** - テストが通る状態を維持しながらコードを改善する

### テストコードの不可侵性

- **テストコードを修正・削除してテストを通すことは絶対に禁止**
- テストが失敗した場合、実装コードが間違っている
- テストは仕様を定義する。実装はその仕様を満たす

## テスト実行ルール

### 最小テストの原則

- 全テストを毎回実行しない
- 修正内容に影響するテストファイル・テストクラスのみを対象として実行する
- CI/CD パイプラインでの全テスト実行は別途行う

```bash
# 良い例: 影響範囲のテストのみ実行
npm test -- --testPathPattern="user-service"
pytest tests/test_user_service.py

# 避けるべき例: 全テストを実行
npm test
pytest
```

## テストカバレッジ

### 必須のテスト観点

| 観点 | 説明 | 例 |
|-----|------|-----|
| 正常系 | 期待通りの入力で期待通りの結果 | 有効なユーザー名でログイン成功 |
| 異常系 | 不正な入力でエラーハンドリング | 空のパスワードでエラー |
| 境界値 | 境界条件での動作 | 最大文字数ちょうど、最大+1 |
| エッジケース | 特殊な入力での動作 | null、空文字、特殊文字 |

### テストの品質

- テスト名は「何をテストしているか」が明確にわかるように命名する
- 1テストケースで1つのアサーションに集中する（複数でも可だが関連性を保つ）
- テスト間の依存関係を作らない（各テストは独立して実行可能にする）

```
悪い例:
  test('test1', () => { ... })

良い例:
  test('ユーザー名が空の場合にバリデーションエラーを返す', () => { ... })
```
