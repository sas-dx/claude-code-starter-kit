# Rules（ルール）ガイド

## Rules とは

Rules は、Claude Code が作業する際に自動的に参照する**プロジェクトのルールブック**です。

CLAUDE.md がプロジェクト全体の「業務マニュアル」だとすると、Rules は**業務ごとの詳細な手順書**に当たります。

### 身近な例えで理解する

学校のテストで例えると：

- **CLAUDE.md** = 校則（全校生徒が守る基本ルール）
- **Rules** = 各教科のテストの注意事項（数学のテストでは関数電卓使用可、英語のテストでは辞書持ち込み不可、など）

校則はいつでも適用されますが、教科ごとの注意事項は「そのテストを受けるとき」だけ必要です。同じように、Rules はファイルパスに応じて「その作業をするとき」だけ適用させることもできます。

## なぜ Rules が必要なのか

### 1. CLAUDE.md を短く保てる

CLAUDE.md に全てのルールを書くと数百行になり、AI の注意力が分散します。Rules に分割することで、CLAUDE.md は200行以内のコンパクトな指示書に保てます。

### 2. ルールが整理される

「Git のルール」「コーディング規約」「セキュリティルール」が1つのファイルに混在すると、読みにくく管理しにくくなります。ファイルを分けることで、各ルールの追加・修正・削除が簡単になります。

### 3. パス別にルールを適用できる

「API のコードを触るときだけ適用するルール」「テストコードを書くときだけ適用するルール」のように、特定のファイルパスに絞ったルール適用ができます。不要なルールが適用されないので、AI の判断精度が上がります。

### 4. カスタムコマンド・サブエージェントと連動する

本リポジトリのカスタムコマンドやサブエージェントは、`.claude/rules/` のルールを参照するように設計されています。

| コンポーネント | 参照するルール |
|-------------|-------------|
| `/code-review` | 02_coding-standards, 04_security, 05_review-process |
| `/dev-issue` | 01_git-workflow, 02_coding-standards, 03_testing |
| `/fix-pr-review` | 05_review-process |
| `/resolve-conflict` | 01_git-workflow |
| `code-reviewer` エージェント | 02_coding-standards, 03_testing, 05_review-process |
| `security-reviewer` エージェント | 04_security |
| `rules-coder` エージェント | 全ルール |

ルールを整備すればするほど、コマンドやエージェントの出力品質が向上します。

## 本リポジトリで提供しているルール

| ファイル | 内容 | 主な対象 |
|---------|------|---------|
| [01_git-workflow.md](01_git-workflow.md) | ブランチ命名、コミットメッセージ、PR 作成のルール | Git 操作全般 |
| [02_coding-standards.md](02_coding-standards.md) | 命名規則、エラーハンドリング、可読性のルール | コード実装全般 |
| [03_testing.md](03_testing.md) | TDD、テスト実行、カバレッジのルール | テスト作成・実行 |
| [04_security.md](04_security.md) | 禁止事項、セキュリティチェックリスト | セキュリティ関連 |
| [05_review-process.md](05_review-process.md) | レビュー観点、指摘レベル、対応ルール | コードレビュー |

### 01_git-workflow - Git ワークフロー

ブランチ戦略（feature/fix/docs/refactor プレフィックス）、コミットメッセージのフォーマット（日本語、Issue 番号含む）、PR 作成のテンプレートを定義しています。

### 02_coding-standards - コーディング規約

DRY / KISS / YAGNI の基本原則、命名規則（キャメルケース、パスカルケース等）、エラーハンドリング、コードの可読性（関数サイズ、ネスト制限、マジックナンバー禁止）を定義しています。

### 03_testing - テストルール

TDD の実践方法、テストコードの不可侵性（テストを改竄して通すことの禁止）、最小テストの原則（影響範囲のテストのみ実行）、テストカバレッジの必須観点を定義しています。

### 04_security - セキュリティルール

絶対にやってはいけないこと（DB 削除、機密情報ハードコーディング等）、実装時のセキュリティチェックリスト（認証・認可、インジェクション対策、データ保護、入力バリデーション）を定義しています。

### 05_review-process - コードレビュールール

レビューの基本姿勢、必須チェック項目、指摘レベルの分類（Critical / Warning / Suggestion）、レビュー指摘への対応ルールを定義しています。

## ルールの作り方

### 基本形式

`.claude/rules/` 配下にマークダウンファイルを作成します。

```
.claude/rules/
└── 06_my-rule.md
```

ファイル名の先頭に連番を付けると整理しやすくなります。

### パス別ルール

特定のファイルパスにのみ適用したいルールには、YAML フロントマターで `paths` を指定します。

```markdown
---
paths:
  - "src/api/**/*.ts"
---

# API 開発ルール
- 全エンドポイントに入力バリデーションを実装する
- 統一されたエラーレスポンス形式を使用する
- レート制限を考慮する
```

この例では、`src/api/` 配下の TypeScript ファイルを扱うときだけルールが適用されます。

### グロブパターンの例

| パターン | マッチするファイル |
|---------|---------------|
| `src/**/*.ts` | src 配下の全 TypeScript ファイル |
| `tests/**/*` | tests 配下の全ファイル |
| `*.config.js` | ルート直下の設定ファイル |
| `src/components/**/*.tsx` | コンポーネントの全 TSX ファイル |

## ルール作成のコツ

- **具体的に書く** - 「良いコードを書く」ではなく「関数は50行以内」のように数値や形式で示す
- **良い例・悪い例を併記する** - AI は具体例があると精度が上がる
- **チェックリスト形式にする** - AI が1つずつ確認できる
- **既存ルールとの重複を避ける** - 同じルールが複数ファイルにあると混乱の原因になる
- **プロジェクトに合わせてカスタマイズする** - 本リポジトリのルールはテンプレートなので、実際のプロジェクトに合わせて修正する
