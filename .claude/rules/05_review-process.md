# コードレビュールール

## レビューの基本姿勢

- **全ての問題を報告する** - エラーを見過ごしたり軽視しない
- **具体的に指摘する** - ファイルパス、行番号、修正コード例を含める
- **建設的に伝える** - 批判ではなく改善提案として提示する
- **ルールに基づく** - 個人の好みではなくプロジェクトルールに基づいて判断する

## レビュー観点

### 必須チェック項目

| 観点 | チェック内容 |
|-----|------------|
| **ロジック** | ビジネスロジックが正確か、エッジケースを考慮しているか |
| **セキュリティ** | 04_security.md のチェックリストに準拠しているか |
| **テスト** | 03_testing.md のルールに準拠したテストが書かれているか |
| **規約** | 02_coding-standards.md のルールに準拠しているか |
| **パフォーマンス** | N+1 クエリ、不要な計算、メモリリークがないか |
| **エラー処理** | エラーが適切にハンドリングされているか |

### レビュー結果の分類

| レベル | 意味 | 対応 |
|-------|------|------|
| 🔴 **Critical** | 本番障害やセキュリティ問題につながる | マージ前に必ず修正 |
| 🟡 **Warning** | 品質上の問題やリスクがある | 修正を強く推奨 |
| 🔵 **Suggestion** | より良い書き方の提案 | 対応は任意 |

## レビュー指摘への対応

### 対応ルール

- 全ての指摘に対応する（明示的にスキップ指示がない限り）
- ルールとレビューコメントが競合する場合はプロジェクトルールを優先する
- 指摘が曖昧な場合は最良の解釈で修正し、解釈内容を明示する

### 対応完了報告

修正完了後は PR に以下の形式でコメントを投稿する：

```markdown
## レビュー指摘対応完了

### [指摘1の要約]
- **対応内容**: [修正の説明]
- **変更ファイル**: [ファイルリスト]

全ての指摘事項に対応完了しました。再レビューをお願いいたします。
```
