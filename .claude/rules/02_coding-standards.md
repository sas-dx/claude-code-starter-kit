# コーディング規約

## 基本原則

### DRY（Don't Repeat Yourself）
- 同じロジックを複数箇所に書かない
- 3回以上繰り返すコードは共通化を検討する

### KISS（Keep It Simple, Stupid）
- 必要以上に複雑な実装をしない
- シンプルで読みやすいコードを優先する

### YAGNI（You Aren't Gonna Need It）
- 現時点で必要のない機能を実装しない
- 将来必要になるかもしれない機能の先回り実装は避ける

## 命名規則

<!-- プロジェクトに合わせてカスタマイズしてください -->

| 対象 | 規則 | 例 |
|-----|------|-----|
| 変数・関数 | キャメルケース | `getUserName`, `isValid` |
| クラス・型 | パスカルケース | `UserService`, `OrderItem` |
| 定数 | アッパースネークケース | `MAX_RETRY_COUNT`, `API_BASE_URL` |
| ファイル名 | ケバブケース | `user-service.ts`, `order-item.py` |

## エラーハンドリング

- エラーを黙って握りつぶさない（空の catch ブロック禁止）
- ユーザー向けのエラーメッセージとシステムログを区別する
- 予期しないエラーは適切にログに出力する

```
悪い例:
  try {
    doSomething();
  } catch (e) {
    // 何もしない
  }

良い例:
  try {
    doSomething();
  } catch (e) {
    logger.error('処理に失敗しました', { error: e, context: '...' });
    throw new AppError('処理に失敗しました。再度お試しください。');
  }
```

## コードの可読性

- 1つの関数は1つの責務に限定する
- 関数の行数は50行以内を目安にする
- ネストは3段階以内に抑える（早期リターンを活用）
- マジックナンバーは定数に置き換える

```
悪い例:
  if (status === 3) { ... }

良い例:
  const STATUS_COMPLETED = 3;
  if (status === STATUS_COMPLETED) { ... }
```

## インポート

- 使用していないインポートは削除する
- インポートの順序を統一する（標準ライブラリ → 外部ライブラリ → プロジェクト内）
